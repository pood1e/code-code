# Egress Gateway

## Responsibility

Istio Ambient owns transparent outbound capture for platform and runtime workloads.

The platform policy layer declares external destinations once, then authorizes source service accounts against those destinations.

## External Fields

- `Gateway/egress-waypoint`: waypoint in the egress namespace for destination policy.
- `ServiceEntry`: generated external destination registry entries.
- `AuthorizationPolicy`: generated `targetRefs` authorization grouped by identical source service-account allowlists.
- `Bundle/code-code-egress-trust-bundle`: trust-manager bundle projected to platform and runtime namespaces.

## Implementation Notes

The durable model is `egress.v1.EgressPolicy` in `ConfigMap/code-code-egress-policy`, key `egress-policy.json`.

`platform-egress-service` compiles all `ExternalAccessSet` entries into a destination-indexed desired state:

- one `ServiceEntry` per destination;
- one `AuthorizationPolicy` per source service-account allowlist group;

The support service can seed preset external rule bundles from YAML, but egressservice still sees only the canonical proto model and owns diff/apply. Route resources generated by the removed route model are deleted by owner label on the next apply.

This removes the old source-service rule expansion and the old proxy/AutoProxy execution path. NetworkPolicy remains the coarse fail-closed guardrail. L7 policy, telemetry, and routing use the Ambient waypoint and Gateway API path.

For a corporate HTTP CONNECT proxy, the official Istio shape is still a TCP `ServiceEntry` for one external destination. Networkservice represents it as an `ExternalRule` with TCP protocol, optional `address_cidr`, and `resolution: NONE`. Istio does not model the final external hosts behind the CONNECT tunnel; those host-level rules belong to the proxy service.

TLS CA distribution and any future MITM behavior are separate concerns from network route policy.
