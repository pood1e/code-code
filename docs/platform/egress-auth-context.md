# Egress Auth Context

## responsibility

`platform-auth-service` owns credential material access and header rewrite
policy execution. `egressauth` is the single shared owner for internal auth
metadata header names, standard HTTP credential header names, replacement modes,
placeholder semantics, and adapter ids.

## implementation notes

- Provider surface binding selects the auth policy through
  `header_rewrite_policy_id` / resolved `auth_policy_id`. The surface does not
  duplicate request or response header allowlists; concrete header names remain
  owned by the auth policy.
- Auth-bound control-plane discovery sends only routing metadata through Envoy;
  the auth header itself is generated by auth-service in the ext_authz allow
  response.
- ext_authz never trusts inbound credential header values from the workload.
  For selected routes, auth-service overwrites the credential header from the
  resolved credential material and removes all auth control-plane headers before
  forwarding upstream.
- Request header injection and downstream response header backfill are separate
  ext_authz outputs. Request headers are returned as `OkHttpResponse.headers`;
  client-visible headers such as `Set-Cookie` are returned as
  `OkHttpResponse.response_headers_to_add`.
- The current Istio provider is `envoyExtAuthzGrpc`; per-policy header
  selection lives in the auth policy catalog and route attachment, not in a
  second hard-coded MeshConfig header list.
- Header mutations are lists, not maps. This preserves repeated response
  headers such as `Set-Cookie`; those entries use append semantics instead of
  overwrite semantics. Response replacement returns `remove_headers` for the
  mutated header names and then adds the sanitized/header-backfilled values.
- Credential selection on the data plane uses Istio source principal plus
  `provider_surface_binding_id`; credential ids and secret names are not sent as
  request headers.
- AgentRun egress resolves auth context by runtime source through
  `agent-session`.
- AgentRun prepare asks support-service for `auth_policy_id`, then asks
  auth-service for the concrete header rewrite rules.
- Envoy removes all auth routing headers before forwarding upstream.
- Credential material is read only by `platform-auth-service`.
- Simple header rewrite policy is YAML-defined in `egressauthpolicy`; complex
  rewrite is dispatched by auth-service adapter id.
